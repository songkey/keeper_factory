## 1. 已确认决策

| # | 决策点 | 结论 |
|---|---|---|
| MC1 | 回信匹配 | **主题 token + 发件人白名单**；v0 不做回复口令 |
| MC2 | 生效语义 | **部分生效**：合法行立即执行；非法行汇总为报错/确认邮件；剩余项继续等待。回复指令为手机邮件客户端简化 |
| MC3 | 批量捷径 | **支持 `all ok`；不支持 `all no`**（一行整批否决风险过大） |
| MC4 | 提醒与超时 | 每 **1 小时**提醒（可配置 `mail.reminder_hours`，支持小数）；**永不自动通过** |

## 2. 发送（SMTP，标准库）

- 每轮 Report、批末待审清单发 multipart 邮件：纯文本 + 简单 HTML（由 markdown 渲染）
- **结果图用 OSS 公网 URL 内嵌**——审批人在邮件客户端里直接看图对比，无需登录机器
- 主题带路由 token：`[KF][loop 012] Report`、`[KF][batch 003] 待审批`。token 是回信匹配的锚点
- 批末待审邮件正文列出每项：序号、类型、principle 摘要、证据图片 URL 对、回复指南

## 3. 接收（IMAP 轮询，标准库）

- **只在批次边界等待审批时轮询**（默认间隔 300 秒），平时不收信——避免常驻监听
- 匹配规则：主题含对应 batch token **且**发件人在审批人白名单内；其余邮件一律忽略（v0 安全边界）

## 4. 简化回复协议（MC2）

核心思路：**待审邮件里给每项编号，回复用编号 + 短词**，手机上不需要输入知识 id。

### 待审邮件正文示例

```
[KF][batch 003] 待审批（3 项知识 + 1 项冲突）

── 待审知识 ──
1. [Pattern Patch] pp_0012「逆光人像：先冻结身份再单独发展主体光影」
   证据：原图 vs 结果 https://oss…/a.jpg https://oss…/b.jpg
2. [Failure Note] fn_0005「多指令导致全图重绘」
3. [Capability Note] cn_0003「edit model 无法局部提亮不串色温」

── 冲突裁决 ──
4. pp_0012 与 pp_0009 适用范围重叠且建议矛盾
   4a = 融合两条    4b = 保留 pp_0012    4c = 保留 pp_0009

── 回复方法（直接回复本邮件）──
每行一个决定：
  1 ok        （通过）
  2 no        （否决）
  3 hold      （存疑，安排复验）
  4a          （冲突选项直接回编号）
全部通过可只回一行：all ok
```

### 解析规则

- 短词别名：`ok` = approve（兼容 `yes`/`通过`），`no` = reject（兼容 `拒绝`），`hold` = dispute（兼容 `?`/`存疑`）
- 冲突项回选项编号（`4a`/`4b`/`4c`），编号即决定
- `all ok` = 剩余未处置项全部通过（**不提供 `all no`**，依据 MC3）
- 编号 -> 知识 id 的映射存于批次汇总记录（`batches/batch_003.json`），解析器据此翻译；完整写法（`pp_0012: approve`）仍兼容，作为精确兜底
- 大小写、多余空白、引用文本（`>` 行及原邮件标记之后的内容）全部宽容处理/剥离
- 非法行（未知编号、格式错误）走部分生效：确认邮件列出已生效项 + 报错行，剩余项继续等待

## 5. 超时策略（MC4）

- **永不自动通过**——审批是安全机制，超时放行等于没有审批
- 无回复满 `mail.reminder_hours`（默认 1 小时）-> 发提醒邮件，之后每个间隔重发，无限等待
- 兜底：`kf approve` 本地命令随时可用，**与邮件回复走同一解析器代码路径**——保证未来 UI 后台复用同一套语义

## 6. config.json（`mail` 段）

```json
{
  "mail": {
    "smtp_host": "…", "smtp_port": 465,
    "imap_host": "…", "imap_port": 993,
    "username": "…", "password_env": "KF_MAIL_PASSWORD",
    "from": "keeper-factory@…",
    "approvers": ["you@…"],
    "poll_interval_seconds": 300,
    "reminder_hours": 1
  }
}
```

## 7. 遗留事项（延后处理，持续跟踪）

- HTML 模板美化（v0 只做最简 markdown 转 HTML）
- 多审批人冲突规则（v0 实际单审批人；若多人，先到的合法回复生效）
- webhook 收信方式（引入邮件服务后替换 IMAP 轮询；解析器不变）
